## 2. §10 六問逐答

### Q1 範圍是否與 D1–D8 一致？有無偷渡 0.6 大改版或 fork？

**一致，無偷渡。** X1（fork）與 X2（假多模態）在 §3.1 找不到任何違規項；F-MED-4 明寫「文案不宣稱模型直讀多模態」符合 D2；§3.3 把整套視覺留給 0.6 符合 D3。F-INT-5 狀態列與 F-MED-2 chip 屬小 UI，可接受。

小張力（見 N4）：§3.3 寫「0.5 僅允許**額度／插話**相關小文案與 chip」，但 F-MED-2 是媒體路徑 chip，字面上不在允許清單內。意圖清楚（§3.1 B 已標「小 UI，非 0.6 大改版」），建議 §3.3 措辭補上媒體路徑 chip 以免日後拿 §3.3 否決 F-MED-2。

### Q2 額度文案是否防止「用總額填分項」回歸？

**防得住，但要補一個邊界定義。** 現行 `QuotaRings` 三環只讀 `productUsage`（`usageFor()`），總額環走 `creditUsagePercent`，兩條資料路徑天然分離；§2 再以「禁止複製總額、禁止 0%」+ T-Billing-1/2 鎖住。通過。

缺口（P1-3）：§2 觸發條件寫「`isUnifiedBillingUser === true` **或** `productUsage` 為空／**缺分項**」，但 T-Billing-2 說有 `GrokBuild` 分項時「不強制顯示主文案」——若 live 回傳**部分**分項（有 GrokBuild、缺 Api），兩條規則互相矛盾。建議定案：**主文案僅在三固定產品全部無資料時顯示**；部分缺項時有資料的顯示 %、缺的顯示 `—` + per-ring tooltip。並新增 **T-Billing-3**（部分分項 mock）鎖住。另建議 T-Billing-2 的 mock 同時給 `isUnifiedBillingUser: true` + 分項資料，才真正測到「分項優先於統一文案」的優先權規則（§2 最後一條）。

### Q3 插話與「立刻改做」是否語意分離、可測？

**分離、可測，且與官方語意對齊。** 兩動詞各自對應官方機制（interjection buffer vs cancel-and-send）；§6 已列「插話不 cancel／立刻改做會 cancel」的 RTL 斷言，用 mock ACP client 斷言呼叫序列即可鎖住。

但插話的**生命週期規格有缺**，是本審查最重要的發現（P1-1）：`queued` 之後有三種結局，計畫只寫了一種——

1. 安全點 drain、進入本回合（計畫已涵蓋）；
2. **回合先結束、buffer 未 drain**——插話變成懸置，官方另有 queue 列概念（見 N3），GUI 端呈現未定義；
3. **被 cancel 捨棄**——使用者先插話再按「立刻改做」，cancel 會捨棄已 queued 的插話；UI 若仍顯示「已插話」就是說謊。

更關鍵的限制：AGENTS.md 已記載 **ACP SDK 1.2.1 `zSessionUpdate` 封閉 union 會在 SDK 層丟棄自訂 session update**。若官方以自訂 update 通知 interjection 送達／消耗，GUI **收不到送達確認**。F-INT-5 的狀態機必須設計成只依賴可觀測訊號：`queued` 回應 + prompt response 的 stopReason 回合邊界；「已插話」文案應寫成「已排入，下一個安全點生效」而非宣稱已送達。§5.1 的 live smoke 應加一項：**確認 interjection 在 session/update 流中以什麼型別呈現、SDK 會不會丟**，再定 F-INT-5 的最終狀態文案。

### Q4 0.5.0 是否仍過大？

**偏大但可行，不強制降波。** 0.5.0 必含 11 個 ID，逐項盤點：F-INT-1/2/3/5 是核心不可動；F-BILL-1/2 是文案+popover（小）；F-RT-2 是一個 env 欄位（極小）；F-RT-1 中型但缺口已證實且有 plugin-cc 前例可抄；F-RT-6 與 F-MED-1/2 皆有現成地基（billing/context 本來就是兩條管線；拖放共用 paste save 管線）。

建議**預先聲明滑點順位**（若插話工程爆量，依序滑 0.5.1，不需回來改計畫）：
1. F-MED-2 的**縮圖預覽**部分（chip 保留，縮圖滑走）
2. F-RT-6
3. F-MED-1

F-INT 與 F-BILL 不得滑（符合 §10 Q4 前提）。

### Q5 驗收是否可用自動化測試鎖定？缺哪些 case？

§6 骨架正確（billing 文案條件、interject request shape、process kill helper、T-Billing-1/2、插話/立刻改做 RTL）。**建議補**：

| 建議 ID | 內容 | 對應 |
| --- | --- | --- |
| T-INT-3 | interject 回 method not found → UI 降級提示，且**不**自動改發 cancel | §5.1 降級路徑目前無測試 |
| T-INT-4 | 插話 `queued` 後回合隨即結束（未 drain）→ UI 呈現符合 P1-1 定義 | 生命週期結局 2 |
| T-INT-5 | 已 queued 插話後按「立刻改做」→ 插話標示為已捨棄，不聲稱已送達 | 生命週期結局 3 |
| T-Billing-3 | 部分分項（僅 GrokBuild）→ 有資料顯示 %、缺項 —、主文案依 P1-3 定案行為 | Q2 缺口 |
| T-RT-2 | spawn env 斷言含 `GROK_CLIENT_VERSION` = app 版本 | F-RT-2 無測試 |
| T-RT-6 | RTL 斷言 billing 與 context 標籤分區、不混用 | F-RT-6 無測試 |
| T-MED-1 | 拖放與 paste 共用 save 管線（副檔名白名單、size gate、temp 目錄一致） | F-MED-1 驗收「與貼圖一致」具體化 |

F-RT-1「不留孤兒」無法純自動化：建議驗收拆成 kill helper 單元測試（可自動）+ 手動 smoke 清單（quit／disconnect／換 executable 後 `tasklist` 無殘留 grok 程序）。

### Q6 安全：interject 誤觸 cancel、權限模式、token 外洩？

- **誤觸 cancel**：以 T-INT 系列斷言插話路徑零 cancel 呼叫即可鎖死；「立刻改做」按鈕語意上必須與插話視覺分離（計畫已規定 F-INT-3）。通過。
- **權限模式**：interjection 走同一 session，不改 approval mode、不繞權限——無新攻擊面。注意 YOLO 模式下插話指令同樣立即獲批，屬既有模式語意，無需新防護。
- **token**：billing 文案不含 token；`GROK_CLIENT_VERSION` 僅版本字串，但會進官方 telemetry user-agent（`xai-grok-telemetry/src/http.rs`）——建議 PRIVACY.md 一句話揭露「向 xAI 服務標識本 App 名稱與版本」（透明性 nit，非風險）。
- 0.5.1 的 F-TOOL-1 headless 旁路屬安全敏感（繞 ACP 權限流），落實時必須顯式使用者觸發、不可預設開啟——先記在此，0.5.1 開工時列入該波驗收。
